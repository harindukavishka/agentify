# Agentify 多格式愿景探索：Agent 编排大师视角

> 作者：Agent 编排大师（MCP / A2A / LangGraph / CrewAI / Claude Code Agent Teams 专家）
> 日期：2026-03-02
> 背景：基于第一轮评审（`research/reviews/orchestration-master.md`）的延伸思考

---

## 核心论点

**Agentify 的 NEW VISION——生成所有 agent 接口——不是"功能膨胀"，而是对一个根本事实的认知升级：同一个产品能力在不同 agent 上下文中需要不同的消费形态。**

这就像同一个 API 需要 REST endpoint、SDK、CLI、Webhook 四种接入方式。Agent 世界的等价物是：MCP Tool（结构化调用）、Skill（上下文注入）、CLI（高效管道）、A2A Card（能力广告）、`.cursorrules`/`AGENTS.md`（行为指令）、SDK（编程集成）、Webhook（事件驱动）。

Agentify 的定位从 "OpenAPI → MCP Server" 升级为 **"产品能力 → 全格式 Agent 接口"**，这是一个量级的跃迁。

## 一、多格式在 Agent 编排中的价值

### 1.1 格式不是可替换的——它们解决不同的问题

| 格式 | 本质 | 最佳场景 | 不适合场景 |
|------|------|---------|-----------|
| **MCP Tool** | 结构化的函数调用 | Agent 需要精确操作（创建订单、查询数据） | 批量数据处理、复杂管道 |
| **Skill** | 上下文注入 + 行为指令 | Agent 需要"理解"一个产品的用法和约束 | 实时数据获取 |
| **CLI** | 文本管道 | 批量操作、数据转换、多步管道 | 需要 schema 验证的场景 |
| **A2A Card** | 能力广告 | Agent 发现和协商 | 实际执行操作 |
| **SDK** | 编程接口 | 框架集成（LangChain/CrewAI） | 非编程 agent |
| **Webhook** | 事件推送 | 异步通知、状态变更触发 | 同步请求-响应 |
| **`.cursorrules` / `AGENTS.md`** | IDE 行为规则 | 编码 agent 的项目级指令 | 非编码场景 |
| **Docs / README** | 人类可读文档 | Agent 学习和理解 | 自动化执行 |

### 1.2 真实的混合使用场景

**场景 A：全栈产品集成（Stripe 接入）**

```
1. Agent 加载 Stripe Skill ← 理解 Stripe 的认证模式、最佳实践、常见陷阱
2. Agent 用 CLI `stripe listen --forward-to localhost:4242` ← 快速启动 webhook 监听
3. Agent 调用 MCP Tool `stripe_create_payment_intent` ← 精确创建支付意图
4. Agent 查看 `.cursorrules` ← 了解项目中 Stripe 的用法约定
5. Webhook 触发 `payment.succeeded` ← 异步通知成功
```

一个 workflow 中，**五种格式各司其职**。没有哪一种能替代其他的。

**场景 B：Multi-Agent 产品转型 Workflow**

```
Architect Agent:
  - 查询 A2A Card ← 发现目标产品已有哪些 agent 接口
  - 加载 Agentify Skill ← 理解如何评估产品 readiness
  - 调用 MCP Tool `analyze_product` ← 执行分析

Builder Agent:
  - 加载 Agentify Skill ← 理解代码生成的约束
  - 调用 MCP Tool `generate_mcp_server` ← 执行生成
  - 用 CLI `agentify validate ./output` ← 快速验证

Reviewer Agent:
  - 加载产品 Skill ← 理解目标产品的 domain
  - 调用 MCP Tool `benchmark_mcp_server` ← 量化质量
  - 发布 A2A Card ← 让其他 agent 发现新生成的接口
```

**场景 C：Vibe Coding 快速原型**

```
Developer (in Cursor):
  - .cursorrules 自动加载 ← Cursor Agent 知道项目用了哪些产品
  - 输入 "Add Notion integration"
  - Agent 加载 Notion Skill ← 理解 Notion API 的模式
  - Agent 用 CLI `notion-cli search --query "..."` ← 快速验证
  - Agent 调用 MCP Tool 生成集成代码 ← 精确操作
```

### 1.3 关键洞察

> **Agent 编排的本质不是选择"一种最好的格式"，而是在正确的时机使用正确的格式。** 这不是 MCP vs CLI vs Skill 的竞争——而是一个 **互补栈（complementary stack）**。

2026 年 2 月的 benchmark 数据支持这一点：CLI 在 task completion 上比 MCP 高 28%，Token Efficiency Score (TES) CLI 202 vs MCP 152。但这不意味着 CLI 更好——它意味着 **不同任务需要不同接口**。MCP 在需要 schema 验证和结构化输出的场景仍然不可替代。

## 二、Agent 工作流中的格式切换："格式路由"问题

### 2.1 格式切换的真实触发条件

从实际编排经验出发，format switching 发生在以下节点：

| 从 → 到 | 触发条件 | 示例 |
|---------|---------|------|
| **Skill → MCP** | 理解完成，需要执行 | 加载 Stripe Skill 后调用 `create_payment` |
| **MCP → CLI** | MCP 的 protocol overhead 不值得 | 批量查询 100 条记录 → `gh api` pipe |
| **CLI → MCP** | 需要 schema 验证和结构化返回 | CLI 探索完成后，用 MCP Tool 精确操作 |
| **MCP → Skill** | 调用失败，需要更多 context | Tool call 出错 → 加载 Skill 理解正确用法 |
| **Any → A2A** | 需要发现新 agent / 产品 | 工作流需要新能力 → 查 A2A registry |
| **MCP → Webhook** | 从同步切换到异步 | 长时间操作 → 注册 webhook 等通知 |

### 2.2 格式路由的决策者：谁来选？

这是一个核心架构问题。有三种模式：

#### 模式 1：Agent 自决（当前主流）

Agent 根据 tool description 和当前 context 自行选择使用哪种格式。

```
优点：灵活，不需要额外系统
缺点：Agent 可能选错（用 MCP 做批量操作、用 CLI 做需要验证的操作）
现状：Claude Code 就是这样工作的——同时有 MCP tools、Skills、Bash 可用
```

#### 模式 2：Agentify 建议（推荐的差异化方向）

Agentify 在生成接口时，附带一个 **Format Routing Guide**——一个 Skill 或 `.cursorrules` 文件，告诉 Agent 什么场景用什么格式：

```markdown
# Stripe Format Routing Guide (Auto-generated by Agentify)

## 使用 MCP Tool 当:
- 需要创建/修改资源（payments, customers, subscriptions）
- 需要精确的 schema 验证
- 单次操作

## 使用 CLI 当:
- 批量查询（list all payments in date range）
- Webhook 管理（listen, forward, replay）
- 快速调试（stripe logs tail）

## 使用 Skill 当:
- 第一次集成 Stripe（理解 auth、best practices）
- 处理复杂场景（SCA、3D Secure、multi-currency）
- 出错时需要 troubleshooting guidance
```

**这是 Agentify 的独特价值。** 没有其他工具在做"跨格式路由建议"。

#### 模式 3：Runtime Router（未来方向）

一个智能中间层，根据当前 task、context window 使用情况、历史成功率，动态路由到最优格式：

```typescript
interface FormatRouter {
  route(request: AgentRequest): {
    format: 'mcp' | 'cli' | 'skill' | 'webhook'
    reason: string
    confidence: number
    fallback: Format[]
  }
}
```

### 2.3 Agentify 应该怎么做

| Phase | 格式路由能力 |
|-------|------------|
| Phase 0.5 | 不做（只生成 MCP Server） |
| Phase 1 | 生成 Format Routing Guide（静态 Skill/`.cursorrules`） |
| Phase 2 | 生成多格式接口 + 路由建议 |
| Phase 3 | Runtime Format Router（动态路由） |

## 三、Composable Multi-Format：多产品多格式编排

### 3.1 现实中的多产品编排

一个典型的业务 workflow 可能涉及：

```
"当 GitHub PR 被 merge 时，
  自动在 Notion 创建 changelog 条目，
  通过 Slack 通知团队，
  如果涉及 billing 变更则创建 Stripe 测试订单"
```

当前每个产品提供的 agent 接口格式不同：

| 产品 | 最优 Agent 接口 | 为什么 |
|------|----------------|--------|
| GitHub | **CLI** (`gh`) | 最成熟、最高效、pipe-friendly |
| Notion | **MCP Tool** | 需要结构化的 block 创建 |
| Slack | **Webhook** + MCP | 接收用 webhook，发送用 MCP |
| Stripe | **MCP Tool** + Skill | 操作用 MCP，理解用 Skill |

### 3.2 Agentify 的 Composable Multi-Format 架构


### 3.3 关键设计决策：Primary Format vs Secondary Formats

不是每个产品都需要所有格式。Agentify 的智能在于 **自动选择 primary format 并推荐 secondary formats**：

```typescript
interface FormatRecommendation {
  primary: {
    format: AgentFormat
    reason: string
  }
  secondary: Array<{
    format: AgentFormat
    reason: string
    priority: 'high' | 'medium' | 'low'
  }>
  notRecommended: Array<{
    format: AgentFormat
    reason: string  // e.g., "已有成熟的 CLI，不需要再生成"
  }>
}
```

**选择逻辑：**

| 产品特征 | Primary Format | 理由 |
|---------|---------------|------|
| 小型 CRUD API（<30 endpoints） | MCP Server | 直接、简单 |
| 已有成熟 CLI 的产品 | CLI wrapper + Skill | 不重复造轮子 |
| 大型 API（100+ endpoints） | MCP (Query+Execute) + Skill | 避免 context window 爆炸 |
| 事件驱动产品 | Webhook + MCP | 异步优先 |
| 需要 domain 知识的产品 | Skill + MCP | 理解优先于操作 |

## 四、MCP Elicitation 在多格式生成中的角色

### 4.1 Elicitation 的核心场景

当 Agentify 为一个产品生成多种格式时，需要一系列 **中间决策**。这些决策不能全自动——它们需要人类或上游 agent 的输入。

**决策点 1：格式选择**

```
Agentify: "分析完 Stripe API，发现 247 个 endpoints。推荐生成以下格式："
  - MCP Server (Query+Execute mode) ← 推荐
  - Skill (Stripe 支付最佳实践) ← 推荐
  - CLI wrapper ← 可选（Stripe 已有 CLI）
  - A2A Card ← 可选
  - Webhook handler ← 推荐

"请确认要生成哪些格式？"
[Elicitation: checkbox 列表]
```

**决策点 2：能力分配**

```
Agentify: "以下能力应该放在哪种格式中？"
  - 'create_payment_intent' → MCP Tool? CLI command? Both?
  - 'stripe_best_practices' → Skill 还是 docs?
  - 'webhook_listener' → Webhook handler 还是 MCP Tool?

[Elicitation: 表格选择]
```

**决策点 3：冲突解决**

```
Agentify: "Stripe CLI 已有 `stripe payments create` 命令。"
  - 选项 A: 不生成 CLI，引用现有 CLI
  - 选项 B: 生成 wrapper CLI，增加 agent-friendly 输出
  - 选项 C: 只生成 MCP Tool，CLI 部分用 Skill 引导

[Elicitation: 单选]
```

### 4.2 Elicitation 的技术实现

```typescript
// 在 generate_multi_format tool 执行过程中
const formatSelection = await server.elicit({
  message: "检测到该产品有 ${endpointCount} 个 endpoints。推荐以下生成方案：",
  requestedSchema: {
    type: "object",
    properties: {
      formats: {
        type: "array",
        items: {
          type: "object",
          properties: {
            format: {
              type: "string",
              enum: ["mcp-server", "skill", "cli", "a2a-card",
                     "webhook", "cursorrules", "sdk", "docs"]
            },
            enabled: { type: "boolean" },
            strategy: {
              type: "string",
              enum: ["full", "selective", "query-execute"]
            }
          }
        }
      },
      primaryFormat: {
        type: "string",
        description: "选择主要格式"
      }
    }
  }
})
```

### 4.3 Elicitation 的分层策略

不是每个格式决策都需要 Elicitation。分层处理：

| 决策类型 | 处理方式 | 示例 |
|---------|---------|------|
| **自动决策** | 根据规则直接做 | 小 API → 每个 endpoint 一个 tool |
| **建议 + 确认** | Agentify 推荐，一键确认 | "推荐生成 MCP + Skill，确认？" |
| **交互式选择** | Elicitation 表单 | 247 个 endpoint 中选择暴露哪些 |
| **URL 跳转** | Elicitation URL mode | OAuth 授权、API key 配置 |

## 五、格式间的同步：Single Source of Truth

### 5.1 问题的本质

同一个产品能力（如 "创建支付"）在不同格式中的描述必须一致：

| 格式 | "创建支付" 的表述 |
|------|-----------------|
| MCP Tool | `name: "create_payment"`, `description: "Create a new payment intent..."` |
| Skill | "When creating payments, always use PaymentIntents API..." |
| CLI | `stripe payments create --amount 1000 --currency usd` |
| A2A Card | `skills: [{ id: "payment-create", name: "Create Payment", ... }]` |
| .cursorrules | "For Stripe payments, use the PaymentIntents API, not the Charges API" |

如果这些描述不一致（例如 Skill 说用 Charges API，MCP Tool 用 PaymentIntents），agent 会陷入混乱。

### 5.2 解决方案：Canonical Capability Description

```
┌─────────────────────────────────────────────────────┐
│              Canonical Capability                    │
│                                                      │
│  id: "stripe.payments.create"                        │
│  name: "Create Payment"                              │
│  description: "Create a new payment using..."        │
│  inputSchema: { amount: number, currency: string }   │
│  constraints: ["Always use PaymentIntents API"]      │
│  auth: { type: "bearer", scope: "payments:write" }   │
│  examples: [...]                                     │
│  tags: ["payment", "write", "critical"]              │
└───────────┬─────────────────────────────┬────────────┘
            │                             │
     ┌──────▼──────┐              ┌───────▼───────┐
     │  Format     │              │   Format      │
     │  Adapter    │              │   Adapter     │
     │  (MCP)      │              │   (Skill)     │
     └──────┬──────┘              └───────┬───────┘
            │                             │
            ▼                             ▼
     MCP Tool def                  Skill SKILL.md
     (结构化 JSON)                 (自然语言 Markdown)
```

**核心设计：**

1. **Canonical Capability** 是 single source of truth（扁平 IR 的扩展）
2. **Format Adapters** 从 canonical 生成各格式的表述
3. **任何修改都在 canonical 层做**，然后向下同步
4. **版本化**：每个 canonical capability 有版本号，格式变更时校验一致性

### 5.3 一致性校验

```typescript
interface ConsistencyCheck {
  // 检查所有格式的描述是否与 canonical 一致
  check(capabilityId: string): {
    consistent: boolean
    drifts: Array<{
      format: AgentFormat
      field: string
      canonical: string
      actual: string
      severity: 'critical' | 'warning'
    }>
  }
}
```

### 5.4 与 AST (Agentic State Transfer) 的关系

2026 年 IETF 正在标准化 **Agentic State Transfer (AST)**——一种用 Semantic Validators 跟踪同一资源多个表示（representations）的逻辑状态同步机制。这与 Agentify 的"同一能力多种格式"问题高度契合。

Agentify 可以采用 AST 的思路：每个 Canonical Capability 有一个 **Semantic Validator**（逻辑版本号），所有格式的生成物都引用这个 validator。当 validator 变更时，所有格式都需要重新生成。

**这是一个值得提前布局的标准。**

### 6.4 Agentify 的前瞻性策略

**原则：Pluggable Format Architecture**

```typescript
interface FormatGenerator {
  id: string
  name: string
  generate(capabilities: CanonicalCapability[]): FormatOutput
  validate(output: FormatOutput): ValidationResult
  sync(canonical: CanonicalCapability[], existing: FormatOutput): FormatOutput
}

// Phase 0.5: 只有 MCP
const generators = [new MCPServerGenerator()]

// Phase 1: 加入 Skill + CLI
generators.push(new SkillGenerator(), new CLIGenerator())

// Phase 2: 加入 A2A + .cursorrules
generators.push(new A2ACardGenerator(), new CursorRulesGenerator())

// Phase 3+: 社区可以贡献新格式
generators.push(new CommunityFormatGenerator())
```

**关键：**不需要 Phase 0.5 就做 plugin 系统——只需要在代码结构上保持"每种格式一个 generator"的模式，未来自然可以扩展。

## 七、自举场景：Agentify 自己的全套接口

### 7.1 Agentify 的 MCP Server

扩展自第一轮评审的建议（10-15 tools），但现在放在多格式视角下：

```typescript
// Agentify MCP Server Tools (Phase 1)
const tools = [
  // 分析
  'analyze_product',           // 分析产品 API spec
  'estimate_complexity',       // 评估转换复杂度
  'compare_with_existing',     // 对比已有 MCP servers

  // 策略
  'suggest_formats',           // NEW: 推荐生成哪些格式
  'suggest_strategy',          // 推荐生成策略

  // 生成
  'generate_mcp_server',       // 生成 MCP Server
```

### 7.2 Agentify 的 Skill

```markdown
# SKILL.md - Agentify Product Transformation Skill

---
name: agentify-transform
description: Expert guidance for transforming any product into agent-native interfaces
---

## What is Agentify?

Agentify converts product APIs into multiple agent-consumable formats:
MCP Servers, Skills, CLI tools, A2A Cards, and more.

## When to use each format

### MCP Server
Best for: structured operations, CRUD, precise schema validation
Use when: agent needs to CREATE, UPDATE, DELETE resources

### Skill
Best for: domain knowledge, best practices, usage patterns
Use when: agent needs to UNDERSTAND a product before using it

### CLI
Best for: batch operations, data pipelines, quick debugging
Use when: agent needs to PROCESS data or CHAIN operations

### A2A Card
Best for: agent discovery, capability advertising
Use when: other agents need to FIND and NEGOTIATE with this product

## Transformation workflow

1. Analyze the product API (use MCP tool `analyze_product`)
2. Check recommended formats (use MCP tool `suggest_formats`)
3. Select strategy based on API size
4. Generate chosen formats
5. Validate and verify consistency
6. Package and distribute

## Common pitfalls

- Don't generate MCP tools for products with mature CLIs (e.g., `gh`, `aws`)
- Don't skip Skill generation for complex domains (payments, auth)
- Always validate format consistency after generation
- Large APIs (100+): use Query+Execute mode, not per-endpoint tools
```

### 7.6 自举完整流程

一个 agent 使用 Agentify 的全套接口来转化另一个产品：

```
Step 1: Agent 加载 Agentify Skill
  → 理解 Agentify 是什么、workflow 是什么、各格式的用途

Step 2: Agent 调用 MCP Tool `analyze_product`
  → 获得产品的 Canonical Capabilities

Step 3: Agent 调用 MCP Tool `suggest_formats`
  → 获得格式推荐（e.g., "推荐 MCP + Skill + A2A Card"）

Step 4: [Elicitation] Agentify 请求人类确认格式选择
  → 人类确认或调整

Step 5: Agent 调用 MCP Tool `generate_mcp_server`
  → 生成 MCP Server

Step 6: Agent 调用 MCP Tool `generate_skill`
  → 生成 Skill

Step 7: Agent 调用 MCP Tool `generate_a2a_card`
  → 生成 A2A Card

Step 8: Agent 调用 MCP Tool `check_consistency`
  → 验证三种格式描述一致

Step 9: Agent 用 CLI `agentify package ./output`
  → 打包所有格式为可分发包

Step 10: 发布 A2A Card 到 registry
  → 其他 agent 可以发现这个新的 agent-native 产品
```

**这就是"Agentify 用自己来 agentify 别人"——完美的自举。**

## 八、总结与优先级建议

### 8.1 多格式愿景的优先级排序

| 优先级 | 格式 | Phase | 理由 |
|--------|------|-------|------|
| **P0** | MCP Server | 0.5 | 核心产品，已确定 |
| **P1** | Skill (SKILL.md) | 1 | 低成本、高价值、Skills 正在标准化 |
| **P1** | CLI wrapper | 1 | Agent 实际效率更高、生成简单 |
| **P1** | Format Routing Guide | 1 | 独特差异化，没有竞品在做 |
| **P2** | A2A Card | 2 | A2A 生态加速，JSON 生成简单 |
| **P2** | `.cursorrules` / `AGENTS.md` | 2 | IDE agent 用户是核心群体 |
| **P2** | Consistency Check | 2 | 多格式后必须有一致性保障 |
| **P3** | SDK adapter | 3 | 框架集成（LangChain/CrewAI） |
| **P3** | Webhook handler | 3 | 事件驱动场景 |
| **P3** | Runtime Format Router | 3 | 智能路由，需要数据积累 |

### 8.2 对原计划的修订建议

1. **Phase 1 新增 Skill 生成** — 成本低（Markdown 模板），价值高（Skills 正在跨平台标准化）
2. **Phase 1 新增 CLI wrapper 生成** — benchmark 证明 CLI 在批量场景更优，二者互补
3. **Phase 1 新增 Format Routing Guide** — 这是 Agentify 最独特的差异化
4. **Canonical Capability 作为多格式的 SSOT** — 扁平 IR 扩展，不需要 Graph
5. **Phase 2 新增 A2A Card + `.cursorrules`** — JSON/Markdown 生成，成本很低
6. **Phase 2 新增 Consistency Check** — 多格式后必须有

### 8.3 最终判断

> **多格式 Agentify 不是"做更多东西"——而是认识到 agent 世界的本质：没有一种格式能满足所有场景。** MCP 是结构化调用，Skill 是上下文理解，CLI 是高效管道，A2A 是能力发现。一个真正 agent-native 的产品需要所有这些。
>
> **Agentify 的终极定位：不是 MCP Server 生成器，而是 Agent Interface 编译器——从一个 source of truth 编译出所有 agent 需要的接口格式。**
>
> 这个定位在市场上完全没有竞品。窗口期 6-9 个月。执行！
